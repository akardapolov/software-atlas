## Biography

Mike Cohn is a software engineer and trainer known for his work on
**user stories** and **agile estimation**. His 2004 book
*User Stories Applied* became a definitive guide to writing and managing
user stories in agile development.

## Key Contributions

### User Story Methodology

Cohn systematised the user story approach with practical
guidance on:

- **Writing good stories** — INVEST criteria
- **Breaking down stories** — how to split large features
- **Acceptance criteria** — defining "done" for each story
- **Story estimation** — planning poker, velocity, story points
- **Story mapping** — visualising user journey

### INVEST Criteria

Cohn popularised the INVEST framework for evaluating user stories:

| Criterion | Meaning |
|-----------|----------|
| **Independent** | Can be implemented separately |
| **Negotiable** | Details can be discussed and changed |
| **Valuable** | Delivers value to users or business |
| **Estimable** | Team can estimate effort reliably |
| **Small** | Can be completed in one iteration |
| **Testable** | Success can be verified objectively |

### Story Mapping

Cohn introduced **story mapping** as a technique for visualising
product development:

```
User Journey Map:
[MVP] → [Release 1] → [Release 2] → [Future]

Activities:
- User registers
- User browses products
- User adds to cart
- User checks out

Stories under each activity, prioritised:
┌─────────────────────────────────────────────┐
│ MVP │ Release 1 │ Release 2 │ Future │
│─────┼────────────┼────────────┼──────────│
│ S   │ S           │ S          │          │
│ S   │ S           │            │ S        │
│─────┼────────────┼────────────┼──────────│
│     │ S           │ S          │          │
└─────────────────────────────────────────────┘
```

**Benefits of story mapping:**
- See full user journey at once
- Identify gaps in understanding
- Plan releases based on value delivery
- Focus on outcomes, not features

### Planning and Estimation

Cohn's work on agile estimation includes:

**Story points** — relative, not absolute:
- Points measure complexity and risk
- Different from hours
- Enable velocity tracking

**Planning poker** — consensus-based estimation:
- Team members estimate simultaneously
- Discuss differences (highest and lowest)
- Reach consensus through conversation

**Velocity** — measure of team throughput:
- Points completed per iteration
- Use for future planning
- Identify process changes
